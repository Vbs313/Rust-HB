## 一、背景与动机

原有的主循环是一个单一大循环，负责 IPC 读取、状态判断、AI 搜索全部在一个 `loop` 中。存在 6 个架构级问题：

| # | 问题 | 后果 |
|---|------|------|
| 1 | 单一大循环，职责不分 | 改 AI 必须碰 IPC，改 IPC 必须碰主循环 |
| 2 | CPU 密集的 AI 搜索在 `async` 上下文跑 | 阻塞 tokio 线程池 |
| 3 | 不可取消 | Ctrl+C 无法优雅退出 |
| 4 | 固定轮询 (500ms) | 延迟大或 CPU 浪费 |
| 5 | `&mut IpcClient` 独占 | 无法同时读写 |
| 6 | AI 结果被丢弃 | 决策只打日志不执行 |

## 三、各模块详细设计

### 3.1 Supervisor — 生命周期管理

```rust
pub struct Supervisor {
    shutting_down: Arc<AtomicBool>,
    notify: Arc<Notify>,
    handler_installed: Arc<AtomicBool>,
}
```

- Ctrl+C 信号处理器在第一次调用 `is_shutting_down()` 时惰性注册
- `wait_for_shutdown()` 通过 `Notify` 实现 async 等待
- `shutdown_flag()` 返回 `Arc<AtomicBool>` 供 StateMonitor 等子任务使用
- 4 个单元测试覆盖

### 3.2 AsyncIpcClient — 异步 IPC 包装

```rust
pub trait IpcClientOps: Send + Sync {
    async fn get_game_state(&self) -> Result<GameStateData, IpcError>;
    async fn perform_action(&self, action: ActionCommand) -> Result<bool, IpcError>;
    async fn reconnect(&self, timeout: Duration) -> Result<(), IpcError>;
}
```

- 通过 `Arc<Mutex<IpcClient>>` 实现 `&self` 方法的共享访问
- 所有阻塞操作通过 `tokio::task::spawn_blocking` 转移到阻塞线程池
- 提供 `MockIpcClient` 用于测试（支持注入假数据、模拟断开/恢复）

### 3.3 StateMonitor — 游戏状态监视器

```rust
pub enum ConnectionStatus {
    Connected,
    Disconnected { retry_count: u32 },
}
```

- 后台任务持续读取游戏状态，通过 `watch::channel` 广播
- 指数退避重连：`poll_interval * 2^(retry_count-1)`，默认 50ms → 5s
- 日志防刷屏：每 `2^n` 次失败才输出一次
- 3 个单元测试（含 mock 模拟断开/恢复）

### 3.4 ActionExecutor — 双模式动作执行

```
AI 返回 Action
  │
  ├─ IPC available? ──Yes──▶ ipc.perform_action(cmd)
  │                              │
  │                          成功 ◀── 返回 Ok
  │                              │
  │                           失败
  │                              ▼
  └─ InputSim available? ──Yes──▶ execute_via_input_sim()
                                      │
                                   play_card / attack / end_turn / hero_power
                                      │
                                   成功 → Ok
                                   失败 → Err("not calibrated")
```

- `GameStateData` → `Playfield` → `Ai::do_all()` → `ActionCommand` → IPC/InputSim
- InputSim 路径为骨架，具体坐标校准待实现

## 四、解决了原来的 6 个问题

| # | 问题 | 解决方式 |
|---|------|---------|
| 1 | 单一大循环 | 5 个独立模块，各管一事 |
| 2 | AI 阻塞 tokio 线程 | `spawn_blocking` + `select!` 非阻塞等待 |
| 3 | 不可取消 | Ctrl+C → `Supervisor` → 所有组件停止 |
| 4 | 固定轮询 | `StateMonitor` + 可配置间隔（默认 50ms） |
| 5 | `&mut IpcClient` 独占 | `Arc<Mutex<IpcClient>>` 共享访问 |
| 6 | AI 结果被丢弃 | `ActionExecutor` 将决策发回游戏 |
