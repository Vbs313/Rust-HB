## 二、根本原因

**Hearthstone 进程启用了 Windows 10/11 进程缓解策略：**

```
1. Arbitrary Code Guard (ACG) — 阻止 VirtualAllocEx 分配执行页面
2. Control Flow Guard (CFG) — 阻止跳转到非合法目标
3. ProcessSignaturePolicy — 仅允许签名模块中的代码执行
```

这解释了为什么：

- `call_remote(root_fn)` ✅ **工作**：`root_fn` 在已加载的 `mono-2.0-bdwgc.dll` 中
- `call_remote(shellcode_addr)` ❌ **失败**：`shellcode_addr` 在 `VirtualAllocEx` 分配的内存中
- 写入 `.text`/`.data`/覆盖函数 **全部失败**：`mono_class_from_name` 本身可能被 CFG 保护

## 三、可行的替代方案

| 方案                | 复杂度    | 说明                                    |
| ------------------- | --------- | --------------------------------------- |
| DLL 注入 + 命名管道 | 🔴 高     | 编译 DLL → LoadLibrary 注入 → 管道通信  |
| SetWindowsHookEx    | 🔴 高     | 注入 DLL 到目标进程消息队列             |
| 调试进程 API        | 🟡 中     | 以调试器方式附加，可读写内存、设断点    |
| 内核驱动            | 🔴🔴 极高 | 绕过所有用户态保护                      |
| **放弃运行时连接**  | 🟢 **低** | **AI 引擎纯逻辑已完成，不依赖游戏进程** |
