## 二、Rust 重构战略规划

### 2.1 核心架构决策

| 决策项          | 选择                                 | 理由                      |
| --------------- | ------------------------------------ | ------------------------- |
| Rust 目标       | `i686-pc-windows-msvc`               | 目标游戏为 32 位 x86 进程 |
| Windows API     | `windows` crate                      | 官方 MS 维护的 Win32 绑定 |
| 进程内存读写    | 自制模块（基于 `ReadProcessMemory`） | 替代 GreyMagic            |
| Mono 运行时交互 | `mono-rs` 或自制绑定                 | 读取 Unity/Mono 内部对象  |
| 输入模拟        | `windows` crate 的 SendInput         | 替代 Triton.Client        |
| 配置格式        | `serde_json` / `toml`                | 替代 Newtonsoft.Json      |
| 日志            | `log` + `tracing`                    | 替代 log4net              |
| 异步运行时      | `tokio`                              | 协程/异步任务调度         |
| GUI             | 暂定 TUI (ratatui) 或原生 Win32      | WPF 无 Rust 等价物        |
| AI 搜索         | 纯 Rust 实现（零依赖）               | DFS 搜索无外部依赖需求    |
| 卡牌数据库      | Rust 枚举 + `phf` 或构建脚本生成     | 替代 3000+ C# 枚举        |

### 2.2 多项目工作区结构

```toml
[workspace]
members = [
    "crates/core",              # 核心运行时/日志/配置
    "crates/mono-bridge",       # Mono 运行时桥接（替代 GreyMagic）
    "crates/game-api",          # 游戏操作 API（替代 TritonHs）
    "crates/bot-framework",     # Bot/Plugin/Routine 框架
    "crates/silverfish",        # AI 引擎（替代 Silverfish）
    "crates/silverfish-card-db",# 卡牌数据库（构建时生成）
    "crates/silverfish-sim-cards", # 卡牌模拟（构建时生成）
    "crates/plugins",           # 插件系统
    "crates/input-sim",         # 输入模拟（贝塞尔曲线）
    "crates/process-mgr",       # 进程管理
    "crates/auth",              # 认证系统
    "crates/compression",       # LZMA 压缩
    "crates/app",               # 主程序入口（TUI/Win32）
]
```

### 2.3 阶段划分

| 阶段                  | 内容                                                    | 预估工作量 | 里程碑                     |
| --------------------- | ------------------------------------------------------- | ---------- | -------------------------- |
| **P0: 基础设施**      | Rust 工作区搭建、Windows sys32 绑定、内存读写、日志系统 | 1-2 周     | 能读取外部进程内存         |
| **P1: Mono 桥接**     | Mono 运行时读取、类型系统映射、对象遍历                 | 2-3 周     | 能枚举炉石进程中的 C# 对象 |
| **P2: 游戏 API**      | TritonHs 等价实现：实体读取、TAG 映射、操作封装         | 2-3 周     | 能读取炉石完整状态         |
| **P3: 输入模拟**      | 贝塞尔曲线鼠标移动、键盘模拟、Client 操作               | 1 周       | 能点击炉石 UI              |
| **P4: 弹窗处理**      | DialogHandlers 策略模式移植                             | 1 周       | 能自动处理各种弹窗         |
| **P5: Bot 框架**      | IBot/IPlugin/IRoutine 框架、BotManager                  | 2 周       | 能加载运行 Bot             |
| **P6: Silverfish AI** | Playfield、MiniSimulator、MoveGenerator、Behavior       | 4-6 周     | AI 能做出基础决策          |
| **P7: 卡牌系统**      | CardDB + 1000+ 卡牌模拟（代码生成）                     | 4-8 周     | AI 支持所有卡牌            |
| **P8: 插件 + 认证**   | 4 个插件、LZMA、认证、自动更新                          | 2-3 周     | 完整功能可运行             |
| **P9: 主程序**        | TUI/Win32 GUI、进程管理、配置管理                       | 2-3 周     | 替代原版 Hearthbuddy       |

### 2.4 风险与挑战

| 风险                | 等级  | 缓解措施                              |
| ------------------- | ----- | ------------------------------------- |
| Mono 运行时版本差异 | 🔴 高 | 对关键偏移做运行时检测                |
| 1000+ 卡牌模拟移植  | 🔴 高 | 用 Rust 过程宏/构建脚本自动化代码生成 |
| 贝塞尔曲线行为差异  | 🟡 中 | 提供可配置参数，对比原版录屏          |
| 弹窗检测时效性      | 🟡 中 | 轮询 + 事件混合模式                   |
| 认证协议逆向        | 🟡 中 | 保留现有认证逻辑，逐步理解            |
| 32 位交叉编译       | 🟢 低 | `i686-pc-windows-msvc` 目标已成熟     |
| Win32 API 绑定      | 🟢 低 | `windows` crate 覆盖完善              |

### 2.5 与原版对比优势

| 维度     | C# 原版                      | Rust 重写版                   |
| -------- | ---------------------------- | ----------------------------- |
| 内存安全 | ❌ 大量 unsafe P/Invoke      | ✅ 所有权系统 + `unsafe` 隔离 |
| 性能     | ❌ GC 暂停、装箱拆箱         | ✅ 零成本抽象、无 GC          |
| 部署     | ❌ 需要 .NET Framework 4.8.1 | ✅ 静态链接单 exe             |
| 反检测   | ❌ .NET 易被分析             | ✅ 原生二进制更难逆向         |
| 跨平台   | ❌ 仅 Windows                | ✅ 核心 AI 引擎可跨平台       |
| 包管理   | ❌ NuGet + 手动 DLL          | ✅ Cargo 统一管理             |
