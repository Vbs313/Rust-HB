# /reflect — 自动反思提炼规则

## 项目回顾

Hearthbuddy (炉石兄弟) Rust 重写项目，从 C# 代码重构为 Rust。

## 已确认经验规则

### 规则 1: Win32 结构体布局必须精确

**问题**: `MEMORY_BASIC_INFORMATION` 结构体包含了一个在 x86 上不存在的 `PartitionId` 字段，导致后续所有内存区域枚举返回 0。

**教训**: Win32 API 结构体在 x86 和 x64 上的布局不同。`VirtualQueryEx` 返回的 `MEMORY_BASIC_INFORMATION` 在 x86 上没有 `PartitionId`。必须精确匹配目标架构的 C 结构体定义。

**规则**: 使用 `repr(C)` 的 Win32 结构体必须对照 MSDN 文档逐字段验证大小和对齐。在 x86 (i686) 目标上编译时，优先使用 32 位版本的 struct 定义。

### 规则 2: 块读取 + 本地扫描是进程内存扫描的正确模式

**问题**: 逐 4 字节调用 `ReadProcessMemory` 扫描内存耗时太长（500M+ 次调用）。

**解决方案**: 每次读取 64KB 块到本地缓冲区，在本地逐字节扫描。

**规则**: 进程内存扫描采用 `VirtualQueryEx` 枚举区域 → 分块读取 (64KB) → 本地模式匹配的三段式。

### 规则 3: CreateRemoteThread + VirtualAllocEx 被 ACG/CIG 阻止

**问题**: 在 Hearthstone 进程中无法执行外部注入的 shellcode。所有 `VirtualAllocEx` 分配的内存即使通过 `VirtualProtectEx` 修改权限也无法执行。

**解决方案**: 使用 BepInEx (Unity 插件框架) 或 VEH + 硬件断点注入。

**规则**: 反作弊/进程缓解策略 (ACG/CIG) 阻止外部代码执行。与游戏进程通信使用 IPC (命名管道) 而非代码注入。

### 规则 4: Mono 结构偏移是版本相关的

**问题**: MonoClass/MonoImage 的结构字段偏移在不同 Mono 版本/Unity 定制版中不同。

**解决方案**: 运行时动态发现偏移（通过模式扫描），或从已知工作版本提取偏移表。

**规则**: 不要硬编码 Mono 结构偏移。使用偏移表（36 个 int）或运行时发现机制。

### 规则 5: Rust 类型的 Serde 派生需要全链路覆盖

**问题**: 为 Playfield 添加 Serialize 时，链式依赖的所有字段类型都必须实现 Serialize/Deserialize。

**规则**: 为核心数据结构添加 Serde 时，一次性为所有相关类型（Playfield → Minion → Enchantment, Weapon, Action, HandCard 等）添加 derive。

### 规则 6: 测试覆盖率优于猜测

**问题**: 多个偏移猜测试探在无测试验证的情况下浪费了大量时间。

**规则**: 每次实现核心逻辑后立即编写测试。测试不仅验证正确性，也帮助发现边界情况（如 fatigue 计数器不递增、minion_attack 反击不触发）。

## 项目架构决策

1. **AI 引擎 (hb-silverfish-core)**: 已完成，44 测试通过。核心数据流 `Playfield → MoveGenerator → MiniSimulator → Behavior`。
2. **游戏连接**: 通过命名管道 IPC (`hb-ipc`) 与 BepInEx 插件或 Hearthbuddy.exe 通信。
3. **偏移发现**: 从运行中的 Hearthbuddy.exe 内存提取 `OffsetTable36`。
4. **构建目标**: `i686-pc-windows-msvc` (32-bit x86)。

## 下一步规则

1. 偏移提取后，将其写为 Rust 常量数组（不硬编码结构体定义，只硬编码偏移值）。
2. MonoClass 解析使用偏移表动态读取字段，不同版本只更新偏移表。
3. 游戏状态流水线：`Hearthbuddy.exe → IPC → hb-ipc → hb-game-api → Playfield → hb-silverfish-core`。
